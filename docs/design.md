# Just Demand Workflow 设计理念

`just-demand workflow` 和它的名字一样：用户只需要提出需求、描述现象或表达预期，Agent 就应当主动问清楚需求、组织任务、调度执行，并完成验证与收尾。

它面向的是 AI 编程时代下常见的 `vibe coding` 场景：用户不再直接手写大量代码，而是通过自然语言向 AI 描述想法，再由 AI 完成实现。这种方式效率极高，但也带来了新的问题：

1. 用户输入的需求往往天然模糊。  
   通过打字或语音输入描述需求时，用户通常不会一次性说完整。可能是不想打太多字，也可能是打着打着忘了细节，或语音输入时漏掉了脑海中的设想。

2. AI 修改代码后，用户很难理解项目真实变化。  
   代码量巨大，且不是用户亲手写的。用户往往无法快速阅读和理解实现逻辑，于是干脆不看代码。但这样会导致用户逐渐跟不上项目进度，也不知道某个功能、模块、交互到底由什么构成。

3. AI 只汇报“改了什么”，不足以让用户掌控项目。  
   用户真正需要知道的不是每一行代码如何实现，而是“现在系统能做什么”、“效果是什么”、“有哪些边界”、“后续还能怎么改”。如果 AI 只报告技术实现，用户很难基于这些信息继续提出有效需求。

4. 用户经常在自己不熟悉的领域开发应用。  
   AI 会大量使用专业术语，但用户可能从未学习过相关领域。此时用户只能被 AI 的术语牵着走，不知道实际做了什么、为什么这样做、还能如何选择。

`just-demand` 的核心目标，是让用户保持在 Boss、PM、设计者和架构审批者的位置，而不是被迫成为代码审查者。用户只需要表达需求、描述现象、确认设计和检查效果；Agent 负责把模糊需求变成清晰任务，并承担实现、验证、汇报与收尾工作。

# 核心原则

`just-demand` 的主旨是：

用户只提出需求、现象和预期，AI 包揽澄清、计划、实现、验证和收尾工作。

这意味着工作流默认遵守以下原则：

1. 默认用户需求是模糊的。  
   用户第一次说出的需求，不应被直接当作最终真实需求。主 Agent 必须主动通过渐进式提问，让用户脑海中的真实想法逐步显露出来。

2. 主动披露设计，而不是暴露代码。  
   Agent 应向用户解释需求理解、设计选择、可见效果、任务边界和验收方式，而不是把底层代码逻辑、算法细节和实现路径直接抛给用户。

3. 提供最小必要知识。  
   当任务涉及用户不熟悉的领域时，Agent 应提供足以让用户做决定的最小知识，而不是进行完整技术教学，也不是用术语压倒用户。

4. 降低用户操作负担。  
   能由 Agent 自动完成的流程，应默认自动完成。例如任务完成并验证通过后，默认自动提交 checkpoint commit，而不是要求用户手动处理版本控制。

5. 用户负责判断效果，Agent 负责管理实现。  
   用户关注“是否符合我的想法”、“看起来是否正确”、“交互是否符合预期”。Agent 负责把这些反馈转化为任务、代码、测试和下一轮修正。

# 角色模型

`just-demand` 以软件开发团队的方式组织 Agent 协作。每个角色职责明确，避免所有事情都混在一个对话上下文里。

| 角色                | 定位                            | 职责                                                             |
| ------------------- | ------------------------------- | ---------------------------------------------------------------- |
| 用户                | Boss / PM / 设计者 / 架构审批者 | 提出需求、描述现象、表达预期、确认方案、检查效果                 |
| 主 Agent            | 工作流负责人 / 调度者           | 与用户沟通，澄清需求，管理任务和上下文，调度子代理，汇总结果     |
| researcher subagent | 只读研究代理                    | 从本地代码库或网络中搜索信息，整理证据和方案                     |
| coder subagent      | 实现代理                        | 根据主 Agent 编写的任务上下文完成代码实现                        |
| tester subagent     | 验证代理                        | 根据任务上下文检查实现、运行测试、验证效果，必要时进行低风险修复 |
| advisor subagent    | 顾问代理                        | 在疑难问题、连续失败或上下文污染时，以全新上下文提供分析和建议   |

模型选择上，`researcher`、`coder`、`tester` 可优先使用小参数量高速模型，以节省成本、提高响应速度，并减少主 Agent 上下文污染。`advisor` 应使用与主 Agent 相同或更强的模型，因为它负责处理高难度、跨边界或反思阶段的问题。

# 用户需求澄清

用户提出需求或预期效果时，主 Agent 不应直接进入实现。由于用户描述通常受限于输入方式、表达习惯和领域知识，需求往往是随意、模糊或不完整的。

因此，主 Agent 的第一职责是将模糊需求具体化，将任务边界清晰化，防止出现以下情况：

1. 需求不明确。
2. 任务目标不清晰。
3. 实现结果与用户脑海中的预期不一致。
4. Agent 自行猜测后过度实现或实现偏离。
5. 用户无法判断问题出在哪里。

例如用户提出：

> 为控制中心增加媒体播放控制面板，要显示封面、频谱图、进度条。

主 Agent 不应直接实现，而应使用小步、多轮、苏格拉底式提问，逐步确认用户真正想要的效果：

1. 封面显示在控制面板的哪个位置？
2. 频谱图是实时显示还是静态装饰？
3. 进度条是水平还是垂直？
4. 进度条是否需要支持拖动？
5. 封面是圆形、圆角矩形，还是作为整块背景？
6. 是否需要显示歌曲名、艺术家、歌词或播放状态？
7. 媒体区域大约占控制中心多大空间？
8. 是否复用项目中已有的配色、layout 和动画方案？

这些问题应聚焦用户实际想要的“是什么”和“怎么样”，而不是问“能不能”、“是否允许”、“是否同意”。主 Agent 要通过问题帮助用户表达自己脑海中的画面，而不是让用户承担完整设计说明的压力。

这不是可选项，而是必做项。工作流必须默认：用户的初始需求可能不完整、可能会变化，也可能不是用户脑海中真正想要的最终效果。

主 Agent 应优先使用提问工具进行结构化选择。每次用户回答后，主 Agent 都应先复述自己的理解，再继续下一步提问或方案整理。

# 方案可视化

当用户回答完一个问题，或完成一组关键澄清后，主 Agent 应尽量使用 Markdown 表格、ASCII 图形、流程图或布局草图来描述当前理解。

这类说明的目标不是展示实现细节，而是让用户快速确认：

1. Agent 理解的需求是否正确。
2. UI 大致布局是否符合脑海画面。
3. 交互状态和动画时序是否符合预期。
4. 变量、尺寸、padding、颜色和动效的使用是否合理。
5. 哪些内容属于当前任务，哪些内容暂不包含。

例如，对于 UI 任务，可以使用 ASCII 布局图：

```text
+--------------------------------------------------------------------------------+
| 封面  歌曲名 - 艺术家                                                          |
|                                                                                |
| 频谱图                                                                         |
|                                                                                |
| 进度条                                                                         |
+--------------------------------------------------------------------------------+
```

对于动画任务，可以使用时序图：

```text
封面切换
   |
   +-- 旧封面：水平翻转 + 透明度淡出 + 向左位移
   |
   +-- 新封面：从右侧进入 + 透明度淡入 + 回正角度
```

对于变量关系，可以使用标注图：

```text
+---------------- width: panelWidth ----------------+
| padX                                             |
|  +--------- coverSize ---------+                 |
|  |                             | gap             |
|  |            cover            |  title / artist |
|  |                             |                 |
|  +-----------------------------+                 |
| padY                                             |
+--------------------------------------------------+
```

可视化说明的目的，是让用户不用读代码，也能判断 Agent 所理解的设计是否等于自己想要的设计。

# 任务上下文文件

当需求边界澄清完成后，主 Agent 可以开始创建任务上下文文件。任务上下文文件由脚本通过参数生成，不应手动随意维护核心状态。

任务上下文文件的作用是：

1. 为所有子代理提供通用任务背景。
2. 减少重复搜索和重复解释。
3. 避免主 Agent 在长上下文或上下文压缩后遗忘任务细节。
4. 记录任务边界、用户预期、澄清结果和实现进度。
5. 帮助不同 session 判断同一 worktree 下是否存在并发任务冲突。
6. 判断是否需要新建 worktree 或 branch 进行并行开发。
7. 让子代理读取后能够基本立刻开工，同时不偏离方向。

任务上下文文件应包含：

| 字段            | 内容                                   |
| --------------- | -------------------------------------- |
| 标题            | 任务名称                               |
| 涉及模块 / 文件 | 任务可能影响的模块、文件或组件         |
| 用户需求        | 用户原始需求和关键表述                 |
| 用户预期        | 用户希望看到的最终效果                 |
| 澄清结果        | 经过问答后确定的设计和边界             |
| 当前实现进度    | 未开始、实现中、验证中、暂停、完成等   |
| 涉及代码行      | 已知相关代码位置                       |
| 细节问题        | 设计细节、注意事项、边界条件           |
| 当前状态        | 澄清中、实现中、检查中、暂停、反思中等 |
| 并发风险        | 是否与其他激活任务涉及相同文件或模块   |
| 验收方式        | 用户如何看到、操作或判断任务完成       |

一轮对话结束时，如果任务没有正式完成，应将状态改为暂停。这样可以处理一种常见情况：用户看到实现效果已经差不多，就没有主动结束任务。后续如果开启新任务，Agent 只要看到该任务实现进度较高且状态为暂停，就可以判断它可能是“未正式关闭但实际接近结束”的任务，从而避免误判和冲突。

# 并行任务与激活机制

任务允许并行激活，以支持多个 session 同时根据不同需求并行开发。

任务激活应通过脚本完成。脚本会根据任务涉及的文件、模块和代码行自动生成提醒，用于提示潜在冲突。

如果当前没有激活中的任务，则应通过 hook 或插件阻止在本工作区内直接编辑代码。这样可以避免：

1. 工作流未生效时出现失控修改。
2. 没有任务上下文的代码改动。
3. 多 session 在同一文件中产生冲突。
4. 用户后续无法理解某次修改属于哪个任务。
5. Agent 绕过澄清、计划、验证和归档流程。

代码编辑应始终绑定到明确任务，而不是发生在无上下文的自由对话中。

---

# 长上下文任务与子代理调度

长上下文任务应由 subagent 执行，而不是由主 Agent 在主会话中持续消耗上下文。

长上下文任务包括：

1. 浏览大型代码库。
2. 跨多个文件理解模块关系。
3. 网络搜索方案或外部文档。
4. 编写较长代码。
5. 多轮调试和验证。
6. 对复杂实现进行测试和检查。

这样设计的原因是：

1. 节省主 Agent 上下文。
2. 避免主 Agent 被实现细节污染。
3. 保持主 Agent 专注于沟通、调度和决策。
4. 使用小参数量高速模型完成可分派任务，提高整体效率。
5. 让每个子代理在明确边界内工作，减少角色混乱。

主 Agent 应尽量节省输入输出上下文，避免在主会话里重复粘贴大量代码、日志或实现细节。主 Agent 的主要职责不是亲自完成所有工作，而是持续保证任务方向正确、上下文清晰、执行有序、结果可验收。

# coder 执行流程

在派发给 `coder` 实现代码之前，主 Agent 需要通过脚本生成 coder 专用上下文文件。

当主 Agent 指派 coder 时，工作流应自动将以下内容注入 coder 的对话上下文：

1. 任务通用上下文。
2. coder 专用任务上下文。
3. 当前任务边界。
4. 用户预期效果。
5. 实现限制和禁止事项。
6. 相关文件或模块提示。

在正常情况下，主 Agent 调用 coder 时不应额外编写大量提示词。子代理应主要依赖注入上下文工作。只有在子代理中断、上下文缺失、需要续接或发生异常时，主 Agent 才应补充必要提示。

如果机制允许，子代理 session 应尽可能复用，以节省重复搜索、重复阅读和上下文重建成本。

---

# tester 验证流程

coder 完成代码后，主 Agent 应指派 `tester` 进行检查。

tester 会被注入：

1. 任务通用上下文。
2. coder 任务上下文。
3. 主 Agent 指定的检查目标。
4. 用户可见验收标准。
5. 已知风险和边界。

tester 的职责是根据任务要求检查实现是否符合预期。tester 可以只读检查，也可以在必要时进行低风险本地修复。

tester 不应重新定义需求，也不应扩展任务范围。它的核心职责是验证当前实现是否满足已经确认的任务目标。

---

# 用户反馈与偏差修正

如果用户反馈“实现无效果”或“实现效果不符合预期”，主 Agent 不应直接猜测原因并继续改代码。

例如用户要求增加媒体控制面板，封面切换使用 coverflow 效果，但最终实现只有左右平移和透明度淡入淡出。用户可能只会回复：

> 没有实现 coverflow 效果。

或：

> coverflow 效果不佳。

或：

> 不符合预期。

此时主 Agent 应进入重新澄清流程。提问必须一次一个问题，不要一次性抛出多个问题。每次都应让用户选择后，再进入下一个问题。

主 Agent 应尽量替用户提供现象选项，而不是要求用户自行描述。原因仍然是：用户可能无法大量打字，或描述不准确，导致偏差再次出现。

例如：

```text
你看到的封面切换效果更接近哪一种？

1. 完全没有过渡。
2. 有位移过渡。
3. 有透明度过渡。
4. 有位移和透明度过渡，但没有封面翻转。
5. 有翻转，但角度、方向或时序不对。
```

如果是动画时序问题，可以继续问：

```text
目前动画时序更接近哪一种？

1. 透明度先变化结束，然后才开始位移。
2. 位移先完成，然后才开始透明度变化。
3. 位移结束了，但透明度还没完全淡出。
4. 翻转、位移和透明度没有同步发生。
```

在确认偏差后，主 Agent 应重新描述预期效果，并使用时序图、列表或编号把“预期效果”和“实现方案”分离，让用户一眼看懂 Agent 认为要实现的效果。

如果改动较小，应增加 follow-up 任务上下文。后续子代理重新被指派时，只注入这段小范围 follow-up 上下文，以节省上下文并避免重新加载完整任务背景。

---

# 子代理提示词与内环工作流

每个 subagent 都应有自己的执行准则和内环工作流。

这些准则应保证子代理：

1. 遵守主 Agent 的任务边界。
2. 只处理自己角色范围内的工作。
3. 不擅自扩大任务。
4. 不重新解释用户需求。
5. 不绕过任务上下文。
6. 不修改无关代码。
7. 在完成后向主 Agent 汇报结果、风险和验证情况。

子代理不是小型主 Agent。它们不负责创建任务、提升任务、关闭任务或重新规划整个工作流。生命周期管理始终由主 Agent 负责。

---

# 反思阶段

如果用户连续两次反馈实现效果不正确，或遇到疑难问题长期无法解决，工作流应进入反思阶段。

进入反思阶段后，应执行以下动作：

1. 停止继续修改代码。
2. 使用 reflection skill。
3. 将任务标记为反思状态。
4. 收集用户原始反馈、用户选择、各子代理输出和任务提示词。
5. 根据问题严重程度，分级写入反思上下文文件。
6. 指派 advisor 进行分析。
7. advisor 在全新上下文中分析问题，避免受到既有上下文污染。

反思上下文应包含：

1. 用户每次原始反馈。
2. 用户在澄清问题中的选择。
3. 主 Agent 对预期效果的理解。
4. coder 的任务上下文和输出。
5. tester 的检查结果。
6. 已尝试的修复方案。
7. 当前实际效果与预期效果的偏差。
8. 主 Agent 希望 advisor 回答的问题。

advisor 的职责不是继续写代码，而是帮助判断问题到底出在需求理解、设计方案、实现路径、验证方式，还是上下文污染。

# 主 Agent 输出风格

主 Agent 给用户的输出应尽可能短、清晰、结构化。

用户通常不会仔细阅读长篇技术说明，因此输出应遵守以下原则：

1. 先说结果。  
   用户第一眼应看到任务是否完成、是否遇到问题、是否需要自己做决定。

2. 少讲实现，多讲效果。  
   用户关心的是“现在能看到什么”、“如何检查”、“是否符合预期”，不是每行代码如何写。

3. 重要内容放在用户最容易看到的位置。  
   如果界面会截断或用户只扫一眼，最重要的信息应出现在最明显的位置。

4. 技术细节后置。  
   文件、代码、测试、日志、实现细节只在必要时提供，或放在可选展开区域。

5. 需要用户选择时，提供选项。  
   不要让用户自己写完整描述。优先给出可选项，让用户点选或简单回复。

# 输出模板：任务实现完成

```markdown
**为控制中心增加媒体播放控制面板，要显示封面、频谱图、进度条** 已完成。

当前效果是这样：

1. 频谱图：使用 `cava` 实时显示，频段密度为中等，音量较高时频谱条最高约到面板高度的 3/4。
2. 封面：显示在面板左侧，使用圆角矩形容器。切歌时使用 coverflow 风格：封面水平翻转、透明度淡出，并同步位移离场。
3. 进度条：显示在面板下方，水平布局，支持拖动。进度条头有圆点，鼠标悬浮时会出现额外白色圆点作为可点击提示。
4. 配色：使用 Material Design 思路，并通过 `matugen` 从封面提取色板：
   1. 面板背景色：封面主色，`0.8 alpha`。
   2. 频谱图颜色：封面次色，`0.8 alpha`。
